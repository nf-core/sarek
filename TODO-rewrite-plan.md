# Rewrite Implementation Plan

## Target

- **Binary name**: `ironqc`
- **Tools being rewritten**: `SAMTOOLS_STATS` (`samtools stats`), `MOSDEPTH` (`mosdepth`), `GOLEFT_INDEXCOV` (`goleft indexcov` + required `SAMTOOLS_REINDEX_BAM` pre-index step)
- **Pipeline**: `nf-core/sarek` (`https://github.com/nf-core/sarek`, target release `3.8.1`)
- **Architecture**: single-pass BAM/CRAM bundle with shared iterator and per-tool accumulators (`ironqc bundle`), plus standalone subcommands (`ironqc stats`, `ironqc mosdepth`, `ironqc indexcov`)

## Pipeline Invocation Ground Truth (Step 2.1)

### SAMTOOLS_STATS invocation as used by sarek

- Module: `modules/nf-core/samtools/stats/main.nf`
- Subworkflow call site: `subworkflows/local/cram_qc_mosdepth_samtools/main.nf`
- Effective command template:

```bash
samtools stats --threads ${task.cpus} --reference ${fasta} ${cram} > ${prefix}.stats
```

- Prefix is controlled by config selectors in `conf/modules/modules.config` and `conf/modules/markduplicates.config`:
  - `${meta.id}.md.cram`
  - `${meta.id}.recal.cram`
  - `${meta.id}.sorted.cram` (no markduplicates flow)

### MOSDEPTH invocation as used by sarek

- Module: `modules/nf-core/mosdepth/main.nf`
- Subworkflow call site: `subworkflows/local/cram_qc_mosdepth_samtools/main.nf`
- Effective command template:

```bash
mosdepth --threads ${task.cpus} [--by ${bed}] --fasta ${fasta} ${ext_args} ${prefix} ${cram}
```

- Default `ext.args` in `conf/modules/modules.config` for non-WES: `-n --fast-mode --by 500`
- For WES (`params.wes=true`): `ext.args` is empty string
- Guard behavior in module script:
  - Fails if both input `bed` and `ext.args` include `--by`
  - Fails if `--thresholds` is provided without `--by`

### GOLEFT_INDEXCOV invocation as used by sarek

- Subworkflow: `subworkflows/local/bam_variant_calling_indexcov/main.nf`
- Pre-step module: `modules/local/samtools/reindex_bam/main.nf`
  - writes header-only BAM (`${meta.id}.reindex.bam`)
  - writes filtered index only with `samtools view --write-index ... -F 3844 -q 30`
- `ext.args` from `conf/modules/indexcov.config`: `-F 3844 -q 30`
- Effective indexcov command template:

```bash
goleft indexcov --fai ${fai} --directory ${prefix} ${ext_args} ${input_bams}
```

- In current sarek wiring, `prefix` resolves to `indexcov` (from synthetic meta id), so expected output names are under `indexcov/` and include:
  - `indexcov/indexcov-indexcov.ped`
  - `indexcov/indexcov-indexcov.roc`
  - `indexcov/indexcov-indexcov.bed.gz` (+ `.tbi` generated by `tabix`)
  - `indexcov/indexcov-indexcov.html`
  - `indexcov/*.png`

## Shared Data Structures

- **Gene model**: not required for this bundle (no GTF/GFF-dependent algorithms); use reference contig metadata from BAM header and `.fai` only.
- **BAM iteration**: central iterator emits `RecordView` to registered accumulators:
  - `StatsAccumulator` (flag and mapping statistics)
  - `MosdepthAccumulator` (coverage bins/distributions)
  - `IndexcovAccumulator` (index-bin-like normalization stream and sample summary)
- **Interval trees**:
  - fixed-width windows (`--by 500`) for non-WES mosdepth mode
  - optional BED-driven region map for WES/custom modes
  - contig lookup map from `.fai` for deterministic ordering

## Bundling Architecture

```text
BAM/CRAM reader (single pass)
  -> Record router
      -> StatsAccumulator
      -> MosdepthAccumulator
      -> IndexcovAccumulator
  -> Finalizers
      -> samtools-compatible *.stats
      -> mosdepth *.global.dist.txt / *.region.dist.txt / *.summary.txt
      -> indexcov *.ped / *.roc / *.bed.gz / *.html / *.png
```

## Memory Budget Estimation

- **StatsAccumulator**: mostly counters + modest histograms; expected `20-200 MB` per sample.
- **MosdepthAccumulator**:
  - non-WES default (`-n --fast-mode --by 500`): `100-400 MB` typical human WGS.
  - if per-base output enabled: can exceed `1-3 GB` depending contig size and buffering.
- **IndexcovAccumulator**:
  - 16KB bins across human genome and per-sample vectors: `200-800 MB` for larger cohorts if kept in memory.
- **Mate reconciliation / overlap correction map** (if needed for exact mosdepth behavior): `300 MB` to `2 GB` worst-case on highly unsorted/high-depth regions.
- **OOM warnings**:
  - Multi-sample `indexcov` in one process can spike memory; cap cohort size or stream partial aggregates.
  - Avoid unbounded read-name hash maps; enforce bounded eviction and telemetry.

## Risk Areas

- `samtools stats` parity is high risk due many legacy section semantics and subtle formatting rules.
- `mosdepth` float formatting and cumulative coverage behavior can differ by tiny but report-visible deltas.
- `indexcov` plot and HTML parity can consume multiple sessions even when numeric outputs already match.
- Upstream version split (`samtools 1.21` vs `samtools 1.20` in reindex helper) can introduce hidden differences; test both paths explicitly.
- CRAM edge cases depend on reference handling; enforce strict reference-path checks and error messages.

## Notes

- This plan intentionally separates "match upstream exactly" from later performance improvements.
- Reference outputs must be generated before Rust implementation work starts.
- Keep file names and header strings aligned with MultiQC search patterns:
  - samtools content marker: `This file was produced by samtools stats`
  - mosdepth file suffixes: `*.mosdepth.global.dist.txt`, `*.mosdepth.region.dist.txt`, `*.mosdepth.summary.txt`
  - goleft indexcov: `*-indexcov.ped`, `*-indexcov.roc`
- `ironqc bundle` should be architected for cross-pipeline reuse; avoid hard-coding sarek-specific assumptions except in compatibility profile defaults.
