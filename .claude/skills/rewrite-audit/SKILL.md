# Code Quality Audit

Perform a thorough review of the rewritten tool before integration into the pipeline.
This skill uses Opus because it requires judgment about architecture, naming, and
code quality — not just mechanical checks.

## Automated Checks (Run First)

These must all pass before manual review:

```bash
# Formatting
cargo fmt --check

# Linting (warnings are errors)
cargo clippy -- -D warnings

# All tests pass
cargo test --release

# No compiler warnings
cargo build --release 2>&1 | grep -c "warning" | grep -q "^0$"

# Check for known security vulnerabilities in dependencies
cargo audit

# Find unused dependencies in Cargo.toml
cargo machete

# Verify license compatibility and advisory compliance (if deny.toml exists)
if [ -f deny.toml ]; then cargo deny check; fi
```

If any fail, fix them before proceeding to manual review.

## Code Review Checklist

### Error Handling
- [ ] All fallible functions return `anyhow::Result<T>`
- [ ] Errors propagated with `?` operator
- [ ] Context added with `.context("descriptive message")` or `.with_context(|| format!(...))`
- [ ] `anyhow::bail!()` for early returns with error messages
- [ ] `anyhow::ensure!()` for precondition checks
- [ ] **No `unwrap()` or `expect()` in non-test code** without a comment explaining why it's safe
- [ ] Error messages are actionable (tell the user what to do, not just what went wrong)

### Documentation
- [ ] Every source file has `//!` module-level doc comment (2-4 lines explaining purpose)
- [ ] All `pub` structs, fields, functions, methods have `///` doc comments
- [ ] Complex functions include `# Arguments` and `# Returns` sections
- [ ] Inline `//` comments explain **why**, not **what** (the code shows what)
- [ ] Long files use section dividers (`// ===` for major sections, `// ---` for sub-sections)
- [ ] References to upstream tool behavior are documented (e.g., "Matches R's dupRadar behavior")
- [ ] Known upstream bugs being replicated are explicitly documented

### Naming
- [ ] Types/structs: `CamelCase`
- [ ] Functions/methods: `snake_case`
- [ ] Constants: `SCREAMING_SNAKE_CASE`
- [ ] Modules: `snake_case`
- [ ] Names are descriptive (not `x`, `tmp`, `data` — except in very small scopes)
- [ ] No abbreviations that aren't domain-standard

### Code Structure
- [ ] No dead code without `#[allow(dead_code)]` + justification comment
- [ ] No TODO/FIXME/HACK comments remaining (or they're tracked in an issue)
- [ ] No commented-out code blocks
- [ ] No debug `println!` / `eprintln!` / `dbg!` statements
- [ ] No hardcoded file paths
- [ ] Magic numbers extracted to named constants
- [ ] Functions are reasonably sized (< 100 lines preferred, < 200 acceptable with comments)
- [ ] `#[allow(clippy::too_many_arguments)]` only when refactoring would reduce clarity
- [ ] No tests that silently pass on missing output (e.g., `if file.exists() { assert... }` — must assert existence FIRST)
- [ ] No type truncation in numeric conversions (e.g., `value as u8` silently wraps values ≥ 256). Use `try_from()` or explicit range checks.
- [ ] Verify all `as` casts — especially `as u8`, `as u16`, `as i32` — for potential overflow/truncation in production paths

### Types and Derives
- [ ] `#[derive(Debug)]` on all structs
- [ ] `Clone`, `Default`, `Deserialize` added only when needed — minimal derives
- [ ] Numeric conventions consistent: `u64` for counts/positions, `f64` for metrics, `u8` for flags
- [ ] `IndexMap` used only when insertion order matters; `HashMap` for unordered

### Test Coverage
- [ ] Core algorithm has unit tests
- [ ] Edge cases covered (empty input, single record, all duplicates, etc.)
- [ ] Output formatting tested (float precision, NA handling, column order)
- [ ] Integration tests compare against reference data
- [ ] Test naming: `test_<description_in_snake_case>`
- [ ] Tests use `assert_eq!` with descriptive messages
- [ ] Float comparisons use tolerance: `assert!((val - expected).abs() < tolerance)`
- [ ] Integration tests compare full output files against pinned reference outputs
- [ ] Test data must exercise all relevant code paths (not just happy path — include edge cases: empty input, single-end reads, multi-mapped reads)
- [ ] Float comparison tests use explicit tolerance, not exact equality
- [ ] Reference test outputs generated by script (never hand-edited) — the generation script is committed to the repo

## Architecture Review

### Module Structure
- [ ] Module hierarchy makes logical sense
- [ ] No circular dependencies between modules
- [ ] Public API surface is minimal (only expose what's needed)
- [ ] Inter-module access uses `crate::` paths

### Dependencies
- [ ] All dependencies in Cargo.toml are actually used
- [ ] No unnecessary dependencies (check with `cargo machete` if available)
- [ ] Dependencies are pinned to reasonable version ranges
- [ ] No security advisories: `cargo audit` (if installed)
- [ ] Static linking configured for deployment (`rust-htslib` static features)

### Data Correctness
- [ ] If the tool is part of a multi-tool bundle, verify the accumulator pattern is correctly implemented: each module has `process_read()` and `merge()`, merge is associative/commutative for the data structures involved
- [ ] Verify coordinate systems are consistent throughout (0-based half-open for BAM operations, document any conversions)
- [ ] Check that PE mate handling is correct: verify which BAM flags are used for PE/SE detection, how orphaned mates are handled, and whether the mate buffer key includes position (for multi-mapped reads)

### Performance
- [ ] No obvious performance anti-patterns (allocation in hot loops, etc.)
- [ ] BAM file processed in single pass (not re-read for different analyses)
- [ ] Large data structures pre-allocated where size is known
- [ ] I/O is buffered

## Documentation Review

### README.md
- [ ] Describes what the tool does
- [ ] Build instructions (`cargo build --release`)
- [ ] Usage examples with real command lines
- [ ] Benchmark results with test conditions
- [ ] Comparison with upstream tool(s)
- [ ] Known limitations documented
- [ ] Benchmark numbers in README must match `benchmark/README.md` exactly — verify both are current
- [ ] If plots are generated, include example plot images

### AGENTS.md (for AI-assisted development)
- [ ] Build / lint / test commands documented
- [ ] Project structure documented
- [ ] Code style conventions documented
- [ ] Key dependencies listed with purpose
- [ ] CI pipeline described

### Cargo.toml
- [ ] `edition = "2021"` (or current stable)
- [ ] Release profile optimized: `lto = true`, `codegen-units = 1`, `strip = true`, `opt-level = 3`
- [ ] Package metadata filled in (name, version, description, license)

## Output

Write `TODO-rewrite-audit.md` with:
- List of all checks performed
- Issues found and fixes applied
- Any remaining items that need attention
- Overall assessment: ready for integration or needs more work
